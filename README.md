# ParkSense — Bengaluru Parking Enforcement Intelligence

**Gridlock Hackathon 2.0 — Theme: Poor Visibility on Parking-Induced Congestion**

ParkSense turns raw illegal-parking violation data into a ranked, explainable enforcement
plan. Instead of reactive, patrol-based enforcement, it shows traffic authorities **where**
illegal parking is choking traffic right now, **how much** congestion each hotspot is causing,
and **which patrol** to dispatch first — with the expected recovery in traffic flow.

> Problem direction addressed: *"How can AI-driven parking intelligence detect illegal parking
> hotspots and quantify their impact on traffic flow to enable targeted enforcement?"*

---

## Why this matters

Today, enforcement teams face three gaps (from the problem statement):

- Enforcement is patrol-based and **reactive**.
- There is **no heatmap** of parking violations vs. congestion impact.
- It is **difficult to prioritize** enforcement zones.

ParkSense closes all three with a single operational dashboard.

---

## Key features

- **Live hotspot heatmap** — Leaflet map with pulsing markers for the top illegal-parking
  hotspots across Bengaluru, colored by enforcement tier (Red / Amber / Green).
- **Congestion Impact Score (CIS)** — a transparent, weighted score (0–100) that quantifies
  how much each hotspot is hurting traffic flow.
- **Enforcement Priority Queue** — every zone ranked by a priority score that blends urgency
  and patrol feasibility, so teams know the dispatch order. Exportable to CSV.
- **Time simulator** — scrub any hour (00–23) to see how violations, congestion and priority
  change across the day (rush-hour spikes vs. clear nights).
- **Automated Situation Report** — a per-zone natural-language briefing generated by a
  **deterministic rule engine** (no LLM), so every sentence is traceable and reproducible.
- **Dispatch action** — one click estimates the before/after priority drop and commuter-minutes
  saved via the deploy API.
- **Engineering transparency** — a "Show raw payload" toggle reveals the live JSON and the full
  CIS score breakdown for any zone.
- **Live event stream** — a WebSocket feed pushes live enforcement updates every few seconds.

---

## How the Congestion Impact Score (CIS) works

CIS is a weighted blend of five signals computed per hotspot, per hour:

| Signal | Weight | What it captures |
|---|---|---|
| Violation pressure | 0.42 | Weighted volume of violations + lane blockage + dwell time |
| Temporal surge | 0.22 | Time-of-day demand multiplier (rush hours weigh more) |
| Road criticality | 0.18 | How important the road is to the network |
| Spillback risk | 0.13 | Likelihood of queues spilling into adjacent junctions |
| Historical non-compliance | 0.05 | Repeat-offender history for the zone |

The **priority score** then combines CIS-driven *urgency* (0.75) with patrol *feasibility*
(0.25, based on ETA and availability), and maps to enforcement tiers:

- **RED** (> 65): dispatch now
- **AMBER** (45–65): pre-emptive sweep
- **GREEN** (< 45): monitor

Every component is returned in the API (`score_breakdown`) and visible in the UI, so the score
is fully explainable — not a black box.

## Tech stack

- **Backend:** Python, FastAPI, Uvicorn, WebSockets
- **Frontend:** React 18, Vite, Leaflet + leaflet.heat
- **Data:** seeded synthetic violation dataset (1,000 rows) modeling Bengaluru hotspots

## Run it locally

### 1. Backend (terminal 1)

```bash
cd parksense/backend
pip install -r requirements.txt
python -m uvicorn main:app --host 127.0.0.1 --port 8000
```

API base URL: `http://127.0.0.1:8000` — health check: `http://127.0.0.1:8000/health`

### 2. Frontend (terminal 2)

```bash
cd parksense/frontend
npm install
npm run dev
```

UI: `http://localhost:5173`

> If your backend runs on a different host/port, set `VITE_API_BASE` before `npm run dev`
> (Windows: `set VITE_API_BASE=http://127.0.0.1:8000`).

### Production build (optional)

```bash
cd parksense/frontend
npm run build      # outputs to dist/
npm run preview    # serves the built app
```

## API endpoints

| Method | Endpoint | Purpose |
|---|---|---|
| GET | `/health` | Liveness check |
| GET | `/api/v1/dashboard/overview?hour=` | KPIs, zones, ranking, 24h timeline |
| GET | `/api/v1/zones?hour=` | All zones for an hour |
| GET | `/api/v1/zones/{zone_id}?hour=` | Single zone + forecast |
| GET | `/api/v1/patrol-routes/live?hour=` | Patrol availability |
| GET | `/api/v1/congestion-events?window=2h&hour=` | Recent congestion events |
| POST | `/api/v1/enforcement/deploy?hour=` | Simulate dispatch → before/after impact |
| WS | `/ws/live` | Live enforcement event stream |

## Demo flow (3 minutes)

1. Open the dashboard at **18:00** — the top 3 hotspots pulse on the map; Silk Board is the
   #1 RED zone.
2. Click a hotspot → see its **CIS**, lane block, speed forecast, and the **Automated Situation
   Report** with the recommended patrol.
3. Hit **Show raw payload** to prove the score is transparent and rule-based.
4. Move the **time simulator** to 00:00 → most zones turn GREEN, proving the model reacts to
   real demand instead of flagging everything as critical.
5. Click **Dispatch** on the top zone → see the estimated priority drop and commuter-minutes saved.

## Notes on data & "AI"

- The violation dataset is **synthetic but realistic** (seeded, reproducible) and models real
  Bengaluru hotspots, vehicle mix, rush-hour bias, dwell time and lane blockage.
- The situation report is produced by a **deterministic rule engine**, not an LLM — chosen for
  speed, zero cost, and 100% reproducibility for enforcement use. The architecture cleanly
  supports swapping in an LLM for narrative generation later.
